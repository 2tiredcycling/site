# V4.5 骑行服预报名功能设计

## 目标

V4.5 的目标是新增一个可用于骑行服、骑行裤、套装或其他社团装备的预报名模块。

该模块不是商城系统，也不处理线上付款。它用于展示预报名批次、收集成员意向、支持成员查询和取消，并为管理员提供管理与导出能力。

## 功能定位

预报名功能应独立于活动报名功能。

活动报名继续用于具体骑行活动；骑行服预报名用于装备批次的长期或阶段性收集。二者可以复用相似的页面风格、表单交互和重复报名处理思路，但不应共用活动报名的数据表。

## 前台入口

前台导航栏新增入口：

```text
骑行服预定
```

建议放置在“活动”和“路线”之间。

前台建议包含以下页面：

* 预报名批次列表页。
* 预报名详情页。
* 预报名提交页。
* 预报名成功页。
* 预报名查询和取消页。

## 批次设计

每次预报名由管理员手动创建一个批次。批次不强制按年份命名，可以按实际需求命名。

示例：

```text
2026 春季骑行服预报名
社团骑行服第二批预报名
夏季骑行裤预报名
```

批次建议包含以下字段：

* 标题。
* 状态。
* 开始时间。
* 截止时间。
* 价格下限。
* 价格上限。
* 价格说明。
* 展示图片，支持多张。
* 尺码表图片，建议支持至少一张。
* 说明文字。
* 是否在前台展示。
* 创建时间。
* 更新时间。

批次状态包括：

* 即将开始。
* 正在进行。
* 已经结束。

第一版建议由管理员手动切换批次状态，同时系统仍应根据截止时间限制用户取消。

## 价格说明

不需要付款状态，也不需要线上付款流程。

前台应展示价格区间，并说明最终价格由实际人数决定，但不会超过上限。

示例文案：

```text
预计价格：180-220 元。最终价格将根据实际预报名人数确定，但不会超过上限。
```

## 人数显示

预报名页面应显示当前人数。

人数统计建议只统计非取消记录：

* 待确认。
* 已确认。
* 待领取。
* 已领取。

不计入：

* 已取消。

## 尺码提醒

预报名详情页和提交页应明显展示尺码提醒。

建议文案：

```text
尺码偏小。不追求极致贴身的非公路车玩家，建议买大一码或大两码。
```

尺码表图片应和展示图片分开处理，避免用户只看商品图而忽略尺码信息。

## 预报名记录

每条预报名记录应属于一个批次。

建议字段：

* 所属批次。
* 姓名。
* 学号。
* 手机号。
* 性别。
* 尺码。
* 件数。
* 备注。
* 状态。
* 取消时间。
* 创建时间。
* 更新时间。
* 来源 IP。
* User-Agent。

预报名记录状态包括：

* 待确认。
* 已确认。
* 已取消。
* 待领取。
* 已领取。

## 重复提交规则

同一个批次内，同一个学号只能有一条有效预报名记录。

如果用户再次提交相同学号：

* 如果已有非取消记录，应提示用户是否修改原预报名信息。
* 如果用户确认修改，则更新原记录。
* 如果用户取消修改，则不提交新记录。

同名不同学号不应被判定为重复。

## 查询和取消

用户应可以通过姓名和学号查询自己的预报名记录。

用户取消预报名时，不应删除记录，而是将状态改为：

```text
已取消
```

取消规则建议：

* 截止日期前，用户可以自行取消。
* 截止日期后，用户不能自行取消，需要联系管理员处理。
* 已领取记录不可由用户取消。
* 已取消记录不可重复取消。

## 表单字段

第一版前台表单建议包含：

* 姓名，必填。
* 学号，必填。
* 手机号，必填。
* 性别，建议必填。
* 尺码，必填。
* 件数，必填，默认 1。
* 备注，选填。

尺码选项第一版可以先固定为：

```text
XS / S / M / L / XL / 2XL / 3XL / 4XL
```

如果未来不同批次的尺码差异较大，再考虑改为后台可配置。

## 后台管理

后台建议新增以下页面：

* 预报名批次列表。
* 新建预报名批次。
* 编辑预报名批次。
* 批次预报名记录列表。
* 预报名记录详情或编辑页。

后台能力建议包括：

* 创建、编辑、隐藏批次。
* 上传和管理多张展示图片。
* 上传和管理尺码表图片。
* 查看当前预报名人数。
* 查看、筛选、搜索预报名记录。
* 修改预报名记录状态。
* 导出预报名记录。

导出字段应至少包含：

* 姓名。
* 学号。
* 手机号。
* 性别。
* 尺码。
* 件数。
* 状态。
* 备注。
* 创建时间。

## 数据模型建议

建议新增独立模型，不复用活动报名表。

可选命名：

* `KitPreorderBatch`
* `KitPreorderImage`
* `KitPreorderRegistration`

或更通用的命名：

* `MerchPreorderBatch`
* `MerchPreorderImage`
* `MerchPreorderRegistration`

如果未来可能扩展到更多社团装备，建议使用更通用的 `MerchPreorder` 命名。

## 测试建议

需要新增或更新测试：

* 前台批次列表页可访问。
* 批次详情页可访问。
* 用户可以提交预报名。
* 同批次同学号不会创建重复有效记录。
* 同名不同学号可以分别提交。
* 用户可以通过姓名和学号查询记录。
* 截止日期前可以取消。
* 截止日期后不能自行取消。
* 后台批次管理页需要登录和权限。
* 后台可以查看预报名记录。
* 导出功能可生成预期字段。


## 主要风险

* 如果直接复用活动报名模型，后续会导致活动和装备逻辑混杂。
* 如果不设置截止时间限制，用户可能在统计完成后继续取消，影响采购数量。
* 如果不保留取消记录，管理员后期无法追踪人数变化。
* 如果尺码提醒不明显，用户可能按日常衣服尺码购买，产生尺码偏小问题。
* 如果展示图片和尺码表图片混在一起，用户可能忽略尺码表。
* 如果不做导出，最终整理采购名单会比较低效。
* 如果未来需要付款、库存或多商品购物车，该模块可能需要进一步升级，不建议第一版一次性做成完整商城。
