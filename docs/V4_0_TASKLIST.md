# V4.0 详细任务单（范围冻结与架构落地）

适用版本：`V4.0`

更新时间：2026-03-19  
版本目标：完成 V4 的范围冻结、信息架构定稿、数据模型与迁移方案落地，为 V4.1 前台开发和 V4.2 后台功能开发提供稳定基线。

## 1) 版本范围（V4.0 做什么）

### In Scope（本版本必须完成）
- 明确 V4 范围文档（官网 + GPX 模块边界 + 不做项）
- 完成站点信息架构（页面树、导航结构、核心入口）
- 完成官网新增模块的技术方案设计（前台页面、后台内容管理、活动报名）
- 完成新增数据表与字段设计，并提供可重复执行的迁移脚本
- 明确权限矩阵在 V4 新模块中的映射方案
- 输出 V4.1 / V4.2 的开发输入文档（接口草案、模板草图、验收清单）

### Out of Scope（本版本不做）
- 首页视觉重设计与完整前端实现（放到 V4.1）
- 活动报名完整业务功能上线（放到 V4.2）
- 公告定时发布、媒体库、导出、告警等增强能力（放到 V4.4+）

## 2) 任务拆解（可执行）

## A. 范围与信息架构冻结

- [x] `V4.0-T01` 产出《V4 范围说明》
  - 交付物：`docs/V4_SCOPE.md`（文档头标注 `适用版本：V4.0`）
  - 内容要求：目标、边界、非目标、阶段拆分、里程碑
  - 验收标准：文档包含 P0/P1 映射关系；明确“文化节前必须完成项”

- [x] `V4.0-T02` 产出《站点信息架构（IA）》
  - 交付物：`docs/V4_IA.md`（文档头标注 `适用版本：V4.0`）
  - 内容要求：首页、社团介绍、管理团队、活动、路线、反馈、联系 的页面树与导航
  - 验收标准：每个页面包含入口来源、面包屑/返回路径、权限要求（公开/登录/管理员）

- [x] `V4.0-T03` 定义页面级 URL 规范
  - 交付物：`docs/V4_URL_MAP.md`（文档头标注 `适用版本：V4.0`）
  - 内容要求：公开页面路由、后台路由、API 路由、保留/兼容路由
  - 验收标准：不与现有 `routes_web.py` / `routes_admin.py` 冲突；给出重定向策略

## B. 数据模型与迁移方案

- [x] `V4.0-T04` 完成新增模型设计（不直接上线功能）
  - 交付物：`docs/V4_DATA_MODEL.md`（文档头标注 `适用版本：V4.0`）
  - 覆盖对象建议：
    - `site_pages`（静态页内容）
    - `announcements`（公告）
    - `homepage_sections`（首页配置）
    - `event_registrations`（活动报名）
  - 验收标准：字段、索引、状态机、审计字段（created_at/updated_at/created_by/updated_by）齐全

- [x] `V4.0-T05` 编写 V4 首个迁移脚本（可重复执行）
  - 交付物：`migrations/versions/*_v4_content_and_registration.py`
  - 内容要求：新增表、必要索引、约束、downgrade 可执行
  - 验收标准：
    - 本地 `upgrade -> downgrade -> upgrade` 成功
    - SQLite/PostgreSQL 至少各验证一次（开发环境可用即可）

- [x] `V4.0-T06` 补充 schema 兼容策略
  - 交付物：更新 `app/services.py` 的 `ensure_schema_compat()`
  - 内容要求：对新增关键字段做“存在即跳过，不存在则补齐”
  - 验收标准：老库启动不会因缺字段报错

## C. 权限模型与后台模块设计

- [x] `V4.0-T07` 扩展权限矩阵设计（仅设计与路由守卫规划）

  - 内容要求：

    - 操作级权限：公告发布、页面编辑、报名状态变更、导出

- [x] `V4.0-T08` 设计后台“内容管理”模块结构
  - 交付物：`docs/V4_ADMIN_CONTENT_DESIGN.md`（文档头标注 `适用版本：V4.0`）
  - 内容要求：菜单入口、列表页、编辑页、发布流程、回滚策略
  - 验收标准：与现有 manage 入口结构兼容，且符合当前 RBAC 风格

- [x] `V4.0-T09` 设计“活动报名”后台处理流
  - 交付物：`docs/V4_EVENT_REGISTRATION_FLOW.md`（文档头标注 `适用版本：V4.0`）

## 3) 执行顺序（建议）

1. `T01 -> T02 -> T03`（先冻结范围和 IA）  
2. `T04 -> T05 -> T06`（再锁定数据与迁移）  
3. `T07 -> T08 -> T09`（补齐权限与后台流程）  
4. `T10 -> T11 -> T12`（收敛为可开发、可测试输入）

## 4) V4.0 完成定义（DoD）

- 上述 `T01~T12` 全部完成并经过一次团队评审
- 至少一份迁移脚本完成 `upgrade/downgrade/upgrade` 验证
- V4.1 与 V4.2 可直接开工（前台/后台均有明确输入，不再反复改范围）
- 文档可被新成员直接使用，不依赖口头解释

## 5) 风险与缓解

- 风险：范围持续膨胀，导致 V4.1 开发延期  
  - 缓解：`T01` 明确非目标，新增需求统一进入 V4.4 待办

- 风险：数据模型一次性设计过重  
  - 缓解：V4.0 只做 P0 必需字段，P1 字段预留扩展位

- 风险：权限策略与现有 manage 冲突  
  - 缓解：`T07` 先出矩阵并在开发前做一次路由审计

- 风险：迁移脚本与历史数据库不兼容  
  - 缓解：`T06` 通过 `ensure_schema_compat()` 做兜底

## 6) 建议里程碑（5-7 天）

- Day 1-2：`T01~T03`
- Day 3-4：`T04~T06`
- Day 5：`T07~T09`
- Day 6-7：`T10~T12` + 评审定稿
