# Push 前检查清单（Tag/Push 通用）

适用范围：每次 `git push` 或 `git tag && git push --tags` 之前。

## 1. 工作区与分支

- [ ] `git status --short` 已确认，没有意外文件（临时文件、备份、证书、数据库）。
- [ ] 当前分支正确（通常为 `main`）。
- [ ] 提交内容与本次目标一致（不混入无关改动）。

## 2. 敏感信息与大文件

- [ ] 未提交 `.env`、私钥、证书、账号口令。
- [ ] 新增媒体体积合理（大图已压缩，原始素材不入库）。
- [ ] `.gitignore` 已覆盖临时文件和原始素材目录。

## 3. 功能可用性（最小回归）

- [ ] 关键页面可打开：首页、社团介绍、活动、路线、联系。
- [ ] 关键流程可走通：至少 1 条核心链路（如活动详情 -> 路线详情）。
- [ ] 本次改动相关功能已手测通过（例如主题切换、图片加载、回退逻辑）。

## 4. 自动化校验

- [ ] 测试通过：`python -m pytest -q`
- [ ] 若改到前端模板/CSS/JS，至少做一次浏览器强刷验证（防缓存误判）。

## 5. README 与文档

- [ ] 若本次改动影响“使用方式/路由/部署/目录结构”，README 已更新。
- [ ] 若仅样式微调或内部重构，可不改 README（在提交说明里写明原因）。

## 6. 版本与发布动作

- [ ] 版本号命名正确（如 `v4.2.3`，避免漏写 `v`）。
- [ ] `APP_VERSION`/发布说明与 tag 一致（如项目要求）。
- [ ] 推送顺序正确：先 `main`，再 `tag`。

## 7. 推荐命令（手动执行）

```powershell
git status --short
python -m pytest -q
git push origin main
git tag -a vX.Y.Z -m "release: vX.Y.Z"
git push origin vX.Y.Z
```

## 8. 失败回滚预案（最小）

- [ ] 已知上一个可用 tag（例如 `v4.2.2`）。
- [ ] 若上线异常，优先回退到上个可用 tag 再排查。
